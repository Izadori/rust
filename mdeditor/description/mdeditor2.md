# アプリケーション仕様書 (Ver 2.0)：Markdown Editor (mdeditor)
## 技術レビュー反映版

## 1. 概要と目的
本アプリケーションは、マークダウン形式のドキュメントを記述・編集するためのスタンドアロン型デスクトップエディタです。単なるテキスト表示に留まらず、高度なシンタックスハイライトや複雑なフォーマット（LaTeX数式など）に対応し、**体感的な高速性（Low Latency, High Responsiveness）**とWYSIWYGに近いプレビュー体験を提供することを目的としています。

## 2. 技術スタック (Technology Stack)
| 層 | 技術 | 詳細/変更点 |
| :--- | :--- | :--- |
| **プラットフォーム** | Tauri | クロスプラットフォーム対応のネイティブデスクトップアプリケーション基盤。 |
| **バックエンド (Logic)** | Rust | ファイルI/O、コアMarkdownパーシングロジック、LaTeX事前処理を担当。性能クリティカルなタスクを分離し、UIブロックを防ぎます。 |
| **フロントエンド (UI)** | TypeScript + React | コンポーネントベースでの開発と高度な状態管理（State Management）を実現します。 |
| **改善点** | Web Workers / Dedicated Thread | パース処理やLaTeXレンダリングはメインスレッドから切り離し、非同期実行を行います。 |

## 3. UI/UX仕様 (User Interface / User Experience)
### 3.1 全体レイアウト
画面は垂直方向に二分割され、以下のコンポーネントで構成されます。

*   **左ペイン（編集エリア）：** テキストエディタが配置されます。（全画面幅の約50%）
*   **右ペイン（プレビューエリア）：** Markdownのレンダリング結果が表示されます。（全画面幅の約50%）

### 3.2 エディタ (Left Panel) - 【最適化】
1.  **同期イベント処理:** キー入力イベント（`onInput`）を直接パースに渡すことはしません。必ず**Debounce/Throttle処理**を実装し、ユーザーが一定時間テキスト入力を停止した場合のみ、内容をバックエンドへ送信します。
2.  **シンタックスハイライト:** MarkdownExtra対応のカスタム構文（例: `!!! Note` など）に対して適切な色付けを行います。
3.  **LaTeX数式処理 (Editor Side):** LaTeX形式の数式は、エディタ上でも単なるプレースホルダーまたは特殊な強調表示として識別される必要がありますが、**メインエディタの入力フローをブロックしないよう工夫が必要**です。理想的には、カーソル移動やホバー時など限定的なタイミングでのみ事前レンダリングを行う設計とします。

### 3.3 プレビュー (Right Panel) - 【最適化】
1.  **同期:** 左ペインのDebounced/Throttledデータが更新されるたびに即座にレンダリングを試みます。
2.  **LaTeX処理 (Rendering Core):** LaTeX数式は、Web Workersまたは専用スレッド内で外部ライブラリ（KaTeXなど）を用いて実際に可視化された形で表示します。
3.  **一貫性:** エディタ側でのハイライトとプレビュー側のレンダリングは、使用するMarkdownパーサーとLaTeXエンジンを統一し、可能な限り見た目の不一致を防ぎます。（*理想的には、同じコアライブラリ群を使うべきです*）

## 4. 機能要件 (Functional Requirements)
| ID | 機能名 | 説明 | 詳細/変更点（重要度：高） |
| :--- | :--- | :--- | :--- |
| FR01 | ファイルのオープン | Markdownファイル(`.md`)を指定してロード。 | **[New]** 最初に開かれたファイルの情報をアプリケーション全体で保持し、現在の編集状態を管理します。 |
| FR02 | ファイルの保存 | 現在の状態をファイルシステムに書き出す。 | **[Refined]** 保存前に必ず「変更が未保存です」という警告ダイアログを表示する必要があります。バックエンドから操作を行います。 |
| FR03 | エディタ同期 (Core) | 左ペインの入力→右ペインのプレビューのリアルタイム連携。 | **【最重要】** 以下の非同期処理フローを厳守する：<br>1. `onInput` イベント発生 $\rightarrow$ 2. **Throttle/Debounce (例: 100ms)** $\rightarrow$ 3. 専用Worker/Threadに送信 $\rightarrow$ 4. Workerがパース/LaTeXレンダリング $\rightarrow$ 5. 結果をメインスレッド経由でUI更新。 |
| FR04 | アプリケーション終了 | 安全なアプリケーションシャットダウン。 | **[Refined]** 未保存の変更がある場合、ユーザーに「破棄しますか？」「再開しますか？」という選択肢を提供する確認フローを実装する。 |
| FR05 | 複数ドキュメント管理 (New) | アプリケーションが複数の`.md`ファイルを扱える構造を持つこと。 | **[Added]** 現時点では単一ファイルの編集に焦点を当てますが、将来的にタブ切り替えや複数ファイルへの対応を見越したState Managementを採用します。（これはアーキテクチャ設計の観点からの指摘です） |

## 5. 非機能要件 (Non-functional Requirements)
1.  **パフォーマンス目標:** ドキュメントサイズが最大50KB程度の場合、入力からプレビューの更新までのレイテンシは**100ms未満**であること。
2.  **アーキテクチャ耐性:** リアルタイムなレンダリング処理によるUIフリーズを完全に排除すること。非同期処理（Web Worker/Thread）が必須です。
3.  **堅牢性:** 全てのI/O操作とパースプロセスにおいてエラーハンドリングを行い、アプリケーションの安定性を確保すること。

---
**開発ステップ案 (Updated):**
1.  プロジェクト初期セットアップ (Tauri + React)。
2.  状態管理を設計し、「Source of Truth」（バックエンドがファイルシステム責任を持ち、フロントエンドが編集カーソル/一時変更責任を持つ）を定義する。
3.  Debouncingロジックを含むUI層のイベントハンドラを実装する。
4.  非同期ワーカーを利用したMarkdownパース＆LaTeXレンダリングパイプラインをRust側で構築し、IPC経由で結果を受け取るフローを確立する。
5.  ファイルI/O（開く・保存）とライフサイクル管理（未保存変更警告）を実装する。
