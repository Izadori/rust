# アプリケーション仕様書 (Final Draft)：Markdown Editor (mdeditor)
## 技術レビュー対応版 Ver 3.0 - Gold Standard Specification

### 1. 開発の目標と原則
本アプリケーションは、単なるエディタではなく、**「体感的な速度」と「表現の一貫性」を極限まで高めた知識工学ツール**である。技術選定（Tauri/Rust/React）は維持しつつ、以下の設計原則を厳守する。

*   **パフォーマンス第一:** ユーザーの入力に対するフィードバック遅延 (Latency) は100ms未満が必須。
*   **シングルソース・オブ・トゥルース (SSOT):** データ（Markdownコンテンツとファイルメタデータ）は、特定のコンポーネントではなく、アーキテクチャの中心的なState Store/Service Layerに一元管理する。
*   **レンダリングの一貫性:** エディタ上での視覚的表現とプレビュー画面上のレンダリング結果が、可能な限り同じロジック（理想的には同一コアライブラリ）によって生成されることが必須である。

## 2. 技術スタック (Technology Stack)
| 層 | 技術 | 役割と採用理由 | アーキテクチャ的考慮点 |
| :--- | :--- | :--- | :--- |
| **プラットフォーム** | Tauri | ネイティブデスクトップ環境の提供。OS連携（ファイルダイアログ、メニュー）を担当。 | Rustをメインプロセスに利用し、システムコールやI/Oを行う。 |
| **コアロジック (Backend)** | Rust | 信頼性の高いデータ処理層。ファイル永続化、**Markdownパーシング（WASM推奨）**、LaTeX前処理の実行を担当する。 | パフォーマンスが最も求められる部分。WebAssembly (WASM) を利用してフロントエンドとの境界での再計算を防ぐことを目指す。 |
| **UI/State Management** | TypeScript + React | ユーザーインターフェースの実装と、アプリケーション全体の状態（State）管理を担当する。 | Centralized State Managementライブラリ（例: Zustand）を採用し、Document IDをキーとしたマッピングを行う。 |

## 3. UI/UX仕様 (User Interface / User Experience)
### 3.1 全体レイアウト
*   **構造:** 垂直二分割レイアウト（左エディタ / 右プレビュー）。
*   **コンポーネントの役割分離:** 左エディタは「編集のための場」、右プレビューは「結果を検証する場」として明確に分ける。

### 3.2 エディタ (Left Panel) - 【重点改善箇所】
1.  **推奨エディターコンポーネント:** 標準の`<textarea>`ではなく、CodeMirrorやMonaco Editorなどの高度なカスタマイズが可能なライブラリを採用する。これにより、パフォーマンスと装飾（Decoration）機能を利用可能とする。
2.  **入力処理パイプライン (THE CORE):**
    *   ユーザー入力 $\rightarrow$ `onInput`イベント発生 $\rightarrow$ **Debounce/Throttle (目標: 100ms)** $\rightarrow$ バックエンドサービスに非同期リクエスト送信。
3.  **LaTeX数式 (Editor Side) の扱い:** エディタ上でのレンダリングは「完全な再現」を目指すのではなく、「入力の流れを妨げないための視覚的な支援」に留める。特定の記法（例：`[[math-start]]$...$$[[math-end]]`）としてマークアップし、ホバーまたはフォーカスアウト時にのみ、ミニプレビューを表示する仕組みとする。
4.  **ハイライト:** MarkdownExtraのカスタム構文に対するシンタックスハイライトを実装する。

### 3.3 プレビュー (Right Panel) - 【重点改善箇所】
1.  **レンダリングエンジン:** フロントエンド（TSX/JS）側で実行されるMarkdownパーサーと、LaTeXレンダラーは、**Rustバックエンドが処理した中間表現（ASTまたは抽象構造体）**を元に再構築することが望ましい。これにより、エディタとプレビューの描画ロジック間の「ズレ」を防ぎます。
2.  **LaTeX可視化:** KaTeX (推奨) または MathJaxを使用して、正確な数式レンダリングを行います。この処理は必ずWeb Workerまたは専用スレッド内で行い、メインUIスレッドをブロックさせません。

## 4. 機能要件 (Functional Requirements)
| ID | 機能名 | 説明 | 実装上の要求事項（MUST） |
| :--- | :--- | :--- | :--- |
| FR01 | ファイルのオープン/ロード | `.md`ファイルを読み込み、エディタとプレビューを初期化する。 | **Multi-Document State:** ロードされたファイルは `(UUID, Path, Content)` の構造体としてState Storeに格納される。複数の開いた状態を管理可能であること。 |
| FR02 | ファイルの保存/書き出し | 現在の状態をディスクに永続化する。 | **Dirty State Management:** 変更が未保存（`is_dirty=true`）の場合、必ずユーザーへの確認モーダルをトリガーすること。ファイルパスとUUIDを結びつける。 |
| FR03 | リアルタイム同期コア | 最速の入力からプレビュー反映までのパイプラインを実現する。 | **必須フロー:** `onInput` $\rightarrow$ Throttling $\rightarrow$ Async Worker Call $\rightarrow$ UI Update. パフォーマンス計測が不可欠。 |
| FR04 | アプリケーション終了 | 状態をクリーンに破棄し、アプリケーションをシャットダウンする。 | 未保存の変更がある場合、ユーザーへの警告と明示的な処理（Discard / Save）フローを経ること。 |

## 5. 非機能要件 (Non-functional Requirements)
*   **パフォーマンス目標:** 入力からプレビュー反映までのレイテンシ $\leq 100\text{ms}$ (Max 50KB)。
*   **信頼性:** 全ての非同期処理は、Promise/AwaitやCallbackを用いたエラーハンドリングを徹底し、UIのフリーズを絶対に発生させないこと。
