# アプリケーション仕様書：Markdown Editor (mdeditor)

## 1. 概要と目的
本アプリケーションは、マークダウン形式のドキュメントを記述・編集するためのスタンドアロン型デスクトップエディタです。単にテキストを表示するだけでなく、高度なシンタックスハイライトや複雑なフォーマット（LaTeX数式など）に対応し、WYSIWYGに近いプレビュー体験を提供することを目的とします。

## 2. 技術スタック (Technology Stack)
| 層 | 技術 | 詳細/採用理由 |
| :--- | :--- | :--- |
| **プラットフォーム** | Tauri | クロスプラットフォーム対応のネイティブデスクトップアプリケーションをRustバックエンドで構築するため。Web技術（React/TSX）による開発効率と、Rustによる高性能なシステム処理能力を両立させます。 |
| **バックエンド (Logic)** | Rust | ファイルI/O、Markdownパーシングエンジンとの連携、LaTeXの事前レンダリングなど、パフォーマンスが要求されるコアロジックを担当します。 |
| **フロントエンド (UI)** | TypeScript + React | コンポーネントベースでの開発を実現し、複雑な状態管理（エディタとプレビューの同期）を効率的に行うために採用します。 |

## 3. UI/UX仕様 (User Interface / User Experience)
### 3.1 全体レイアウト
画面は垂直方向に完全に二分割された構造を採用します。
*   **左ペイン（編集エリア）：** テキストエディタが配置されます。（全画面幅の約50%）
*   **右ペイン（プレビューエリア）：** Markdownのレンダリング結果が表示されます。（全画面幅の約50%）

### 3.2 エディタ (Left Panel)
1.  **シンタックスハイライト:** 標準的なMarkdown記法に加えて、特定の拡張構文に対して適切な色付けが必須です。
2.  **高度な機能対応:**
    *   **MarkdownExtra互換性:** `requirements.md`で言及されている特別なマークダウン要素（例：テーブルの特定形式、カスタムブロックなど）をエディタ側で認識し、ハイライトできる必要があります。
    *   **LaTeX数式表示:** Markdown記法内で埋め込まれたLaTeX形式の数式 (`$$...$$` または `$...$`) は、エディタ上でもプレビュー上でも適切に識別され、理想的にはインラインでの可視化（もしくは特殊なプレースホルダー表示）が必要です。

### 3.3 プレビュー (Right Panel)
1.  **レンダリング:** 左ペインで編集された内容と同期して即座にMarkdownをHTMLとしてパースし、ブラウザ標準に近い形でレンダリングします。
2.  **LaTeX処理:** LaTeX数式は、外部ライブラリ（例: KaTeX, MathJax）を用いて実際に可視化された状態で表示する必要があります。

## 4. 機能要件 (Functional Requirements)
| ID | 機能名 | 説明 | 詳細/備考 |
| :--- | :--- | :--- | :--- |
| FR01 | ファイルのオープン | 外部Markdownファイル（`.md`）を指定してエディタにロードする機能。 | 標準的なOSのファイルダイアログを使用します。ファイルの読み込み後、左ペインの内容を更新し、右ペインも再レンダリングする必要があります。 |
| FR02 | ファイルの保存 | 現在開いているドキュメントの内容をファイルとして永続化する機能。 | ユーザーに保存先の選択肢（または既存のファイルの上書き確認）を提供することが推奨されます。バックエンドからファイルシステム操作を行います。 |
| FR03 | エディタ同期 (Core) | 左ペインでのテキスト入力変更が、右ペインへのプレビューレンダリングを即座にトリガーする。 | 最も重要な機能であり、パフォーマンスの観点からも最適化が必要です。タイプイベント（`onInput`）で処理されます。 |
| FR04 | アプリケーション終了 | メニューを通じてアプリケーションを安全に終了させる機能。 | 保存されていない変更がある場合、ユーザーへの確認ダイアログ表示が求められます。 |

## 5. 非機能要件 (Non-functional Requirements)
*   **パフォーマンス:** エディタの入力に対するプレビューの更新遅延は極力最小限であること（体感速度）。
*   **堅牢性:** ファイルI/OやMarkdownパーシングのエラー発生時にも、アプリケーション全体がクラッシュしない設計とする。
